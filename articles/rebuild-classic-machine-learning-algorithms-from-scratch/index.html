What I Learned by Rebuilding Classic Machine Learning Algorithms from Scratch

Published June 10, 2024

  • #machine learning
  • #algorithms
  • #python
  • #data science
  • #engineering

Machine learning libraries make it easy to train models in a few lines of code.

But that convenience often hides the most valuable part of learning: why models behave the way they do.

To better understand the mechanics behind common algorithms, I rebuilt several classic machine learning models entirely from scratch and evaluated them across both regression and classification tasks.

What started as an implementation exercise quickly turned into a much deeper lesson in numerical stability, feature representation, evaluation metrics, and algorithm-data fit.

This article walks through the most interesting engineering and modeling lessons from that process.

Why Rebuild Them from Scratch?

Using libraries like scikit-learn is excellent for productivity, but it abstracts away the implementation details that shape model behavior.

Rebuilding the algorithms forced me to think about:

  • how gradient descent actually converges
  • why loss function scaling matters
  • how distance metrics influence KNN
  • why categorical features drastically change model suitability
  • how evaluation metrics can produce misleading conclusions

The goal was not to outperform production-grade libraries.

It was to build a stronger intuition for how classical models actually work under the hood.

The Algorithms I Implemented

The project included implementations of:

  • Linear Regression
  • Logistic Regression
  • Decision Tree
  • K-Nearest Neighbors (KNN)
  • Naive Bayes

These were tested across multiple datasets covering:

  • continuous regression problems
  • binary classification
  • categorical feature-heavy datasets
  • imbalanced failure prediction tasks

This variety made the differences between algorithms much more visible.

The First Big Lesson: Loss Functions Matter More Than Expected

One of the earliest surprises came from linear regression.

I initially used Sum of Squared Errors (SSE) as the loss function.

This quickly introduced numerical instability issues.

Because SSE scales directly with dataset size, both the loss value and gradient magnitude became unnecessarily large, making the model highly sensitive to learning rate choices.

Switching to Mean Squared Error (MSE) immediately improved training stability.

The performance gain itself was modest, but the optimization process became far more reliable.

This was one of the first moments where the implementation itself taught something deeper than the textbook formula:

the shape of the optimization landscape is only part of the story — the scale matters just as much.

This kind of issue is easy to miss when working only with high-level libraries.

KNN Was Surprisingly Strong for Regression

I expected linear regression to dominate structured datasets like Boston Housing.

Instead, KNN consistently outperformed it.

This was especially clear in the Boston Housing task, where KNN achieved the strongest (R^2).

The result highlighted an important modeling intuition:

local similarity assumptions can outperform global linear assumptions when the feature-target relationship is highly nonlinear.

This also explained why KNN performed consistently well on the wine quality datasets.

Even though KNN is often introduced as a “simple baseline,” implementing it from scratch made its strengths much easier to appreciate.

The model is deceptively powerful when the feature space preserves meaningful neighborhood structure.

Why Decision Trees Struggled in Regression but Excelled in Classification

Decision trees produced one of the most interesting contrasts in the project.

For regression, they often performed poorly and sometimes even produced negative (R^2) scores.

This suggested severe overfitting and weak generalization.

But in classification tasks, especially on discrete datasets, they performed exceptionally well.

This contrast reinforced a key engineering insight:

model suitability is often determined more by data characteristics than by algorithm complexity.

On highly categorical datasets like Mushroom classification, decision trees naturally aligned with the structure of the input space.

The splitting logic matched the data representation almost perfectly.

That fit mattered more than theoretical model sophistication.

Accuracy Can Lie: The Logistic Regression Failure Case

The most valuable lesson from the classification experiments came from imbalanced datasets.

In several Robot Execution Failure tasks, logistic regression reported very high accuracy.

At first glance, the results looked excellent.

But the F1 score was exactly zero.

That immediately revealed what was happening:

the model had learned to predict only the majority class.

This was a strong reminder that:

accuracy alone can completely hide failure modes in imbalanced classification.

Seeing this emerge from my own implementation made the lesson much more concrete than simply reading about it.

From then on, I started treating precision, recall, and F1 score as first-class metrics rather than secondary validation numbers.

The Real Engineering Problems Were Outside the Algorithms

The hardest parts of the project were not the formulas.

They were the surrounding engineering problems:

  • categorical encoding
  • feature normalization
  • memory efficiency
  • distance computation cost
  • class imbalance
  • overflow prevention
  • runtime performance

KNN, for example, was straightforward mathematically, but its brute-force distance computation quickly exposed computational bottlenecks.

This naturally led to ideas like:

  • KD-Tree
  • Ball Tree
  • approximate nearest neighbor indexing

Similarly, Naive Bayes required careful handling of probability underflow and smoothing.

These are the kinds of implementation details that rarely receive enough attention in classroom discussions, but they dominate real-world ML systems.

The Biggest Shift in My ML Intuition

The most important takeaway from this project was a shift in how I think about model selection.

I used to think in terms of:

“which algorithm is stronger?”

Now I think in terms of:

“which assumptions best match the data?”

That is a much more practical engineering mindset.

The experiments made several things very clear:

  • feature types often matter more than model complexity
  • numerical stability can dominate theoretical elegance
  • local structure can beat global assumptions
  • evaluation metrics shape conclusions
  • preprocessing often matters as much as model choice

Rebuilding the algorithms from scratch made these lessons impossible to ignore.

And that deeper intuition is far more valuable than simply knowing how to call .fit().
